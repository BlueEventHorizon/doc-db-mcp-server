# APP-001 doc-db MCP Server 要件定義書

## 概要

チーム開発環境において、ブランチ別にドキュメントのハイブリッド検索インデックスを共有するための MCP Server。
複数の開発者が異なるブランチで並行作業する際、各ブランチのドキュメント内容に基づいた意味検索を提供する。
すべての重い処理（Embedding 生成・BM25 インデックス構築・LLM Rerank）はサーバー側で実行し、API キー管理を一元化する。

## 前提条件

| ID     | 要件 |
|--------|------|
| PRE-01 | 対象リポジトリのルートに `.doc_structure.yaml` が存在すること（`root_dirs` / `target_glob` / `exclude` / `doc_types_map` を定義） |
| PRE-02 | サーバーに OpenAI API キーが設定されていること。`OPENAI_API_DOCDB_KEY` を優先し、未設定時のみ `OPENAI_API_KEY` をフォールバックとして使用する |
| PRE-03 | サーバーに対象リポジトリへの Git アクセス権限が設定されていること（clone / pull が可能） |
| PRE-04 | クライアント（Claude Code）が MCP Server に SSE/HTTP で接続できること |

## 要件一覧

### FNC-001 インデックス構築（build_index）

ブランチ単位でドキュメントをインデックス化する MCP ツールを提供する。

#### 入力パラメータ

| パラメータ | 型 | 必須 | 説明 |
|-----------|-----|------|------|
| branch | string | ✓ | インデックス化するブランチ名 |
| full | boolean | - | true の場合、既存インデックスを破棄して全件再生成する |

#### チャンク分割

| ID     | 要件 |
|--------|------|
| CHK-01 | チャンクは Markdown の見出し境界（H1〜H6）で切り出す。チャンクには文書パスと見出し階層パス（例: `# A > ## B > ### C`）を保持する |
| CHK-02 | 見出しを持たない文書は文書全体を 1 チャンクとして扱う |
| CHK-03 | チャンクの最大サイズは設計書で確定する |

#### インデックス管理

| ID     | 要件 |
|--------|------|
| ST-01  | インデックスはブランチ単位で分離管理する |
| ST-02  | specs のインデックスは `.doc_structure.yaml` の `doc_types_map` で定義される doc_type 単位（`requirement` / `design` / `plan` 等）にさらに分離管理する。rules は単一カテゴリとして扱う |
| ST-03  | インデックスメタデータに build 完了状態（`complete` / `incomplete`）を持ち、失敗チャンクが存在する場合は `incomplete` として保存する |
| ST-04  | 同一ブランチのインデックスを再構築した場合、既存インデックスを上書きする |

#### Embedding

| ID     | 要件 |
|--------|------|
| EMB-01 | Embedding はチャンク単位で生成する |
| EMB-02 | Embedding モデルおよび次元数は設計書で確定する |

### FNC-002 ドキュメント検索（query）

自然言語クエリでドキュメントを検索する MCP ツールを提供する。

#### 入力パラメータ

| パラメータ | 型 | 必須 | 説明 |
|-----------|-----|------|------|
| query | string | ✓ | 検索クエリ（自然言語、または ID・固有名詞を含むテキスト） |
| branch | string | ✓ | 検索対象のブランチ名 |
| mode | string | - | 検索方式。`emb` / `lex` / `hybrid` / `rerank`（デフォルト: `rerank`） |
| top_n | integer | - | 取得結果数の上限（デフォルト: 10） |
| doc_type | string | - | 検索対象の doc_type をカンマ区切りで指定（例: `requirement,design`）。省略時は全 doc_type を対象とする |
| category | string | - | `rules` または `specs`（デフォルト: `specs`） |

#### 検索処理

| ID     | 要件 |
|--------|------|
| LEX-01 | クエリ語の語彙一致に基づいてチャンクスコアを算出する。ID パターン（例: `FNC-001`）や固有名詞の完全一致を高スコアとして扱う。詳細アルゴリズムおよびパラメータは設計書で確定 |
| SC-01  | Embedding スコアと Lexical スコアを統合して候補をソートする。統合方式は設計書で確定 |
| RR-01  | 統合スコア上位の候補を LLM Rerank に渡し、関連度順に並び替える。`mode=rerank` のときのみ実行する。使用モデルは設計書で確定 |
| RR-02  | LLM Rerank が失敗した場合は Rerank をスキップし、統合スコア順の結果を返す（フォールバック） |
| GRP-01 | クエリに ID パターン（`[A-Z]+-\d+`）または固有名詞・識別子が含まれる場合、Hybrid 検索に加えてリポジトリ上の grep（全文検索）を実行し結果を統合する |
| GRP-02 | grep はインデックスの状態に依存しない。インデックスが `incomplete` の場合でも検索漏れを防止する |
| GRP-03 | grep の対象ファイルは `.doc_structure.yaml` の `root_dirs` / `target_glob` / `exclude` に従う |

#### specs と rules の扱い

| ID     | 要件 |
|--------|------|
| OP-01  | specs 検索のデフォルト doc_type は `requirement` / `design` とする |
| OP-02  | rules 検索は doc_type 概念を持たず、rules 配下を単一カテゴリとして検索する |

#### 出力

| ID     | 要件 |
|--------|------|
| OUT-01 | 検索結果には文書パス・見出し階層パス・チャンクテキスト・スコア・スコア内訳（embedding / lexical / rerank）を含める |
| OUT-02 | なぜヒットしたか（どのチャンクのどのスコアで残ったか）を結果に含める |

### FNC-003 インデックス管理

| ID     | 要件 |
|--------|------|
| MNG-01 | ブランチ別のインデックス一覧（ブランチ名・build state・最終更新日時）を取得できること |
| MNG-02 | 指定ブランチのインデックスを削除できること |

## エラーケース

COMMON-REQ-002 エラーハンドリングポリシーに準ずる。外部要因エラーのみキャッチし、内部バグはサイレントフォールバックせず伝播させる。

| 条件 | 動作 |
|------|------|
| 指定ブランチが Git リポジトリに存在しない | エラーを返し処理を中断する |
| PRE-02 の API キーが未設定または無効 | エラーを返し処理を中断する |
| 指定ブランチのインデックスが未構築の状態で query を実行 | インデックス未構築であることをエラーで返す |
| Embedding API 呼び出し失敗（リトライ後も継続） | build_index を `incomplete` 状態で保存し、失敗チャンク情報をエラーに含める |
| LLM Rerank 失敗 | 統合スコア順の結果を返す（処理継続・フォールバック） |

## 外部依存

| ID     | 依存先 |
|--------|--------|
| EXT-01 | OpenAI API — Embedding 生成・LLM Rerank に使用する |
| EXT-02 | Git — ブランチのチェックアウト・ファイル取得 |

## 非機能要件

| ID     | 要件 |
|--------|------|
| NFR-01 | 複数ユーザーの同時クエリに対応すること |
| NFR-02 | インデックスストレージは並行アクセスに耐えること（詳細は設計書で確定） |
| NFR-03 | サーバーは SSE（HTTP）で MCP クライアントと通信すること |
| NFR-04 | API キー・Git 認証情報はサーバー側の環境変数で管理し、クライアントに公開しないこと |
| NFR-05 | サーバーが停止した場合、既存インデックスデータは失われないこと（可用性・データ保護） |
| NFR-06 | サーバーのエラーおよび処理状況をログとして出力すること（運用性） |
| NFR-07 | 稼働率・フェイルオーバー要件はスコープ外とする（内部開発ツールであり外部公開・本番運用を前提としないため） |

## スコープ外（後回し）

- ローカル未コミットファイルのインデックス対応（machine UUID による個別インデックス）
- 検索品質の評価機能（evaluate）
