# 文書記述ルール

**作成日**: 2026-01-01
**目的**: 全文書の一貫性を確保するためのメタルール
**適用範囲**: `docs/rules/`, `docs/specs/`, `.claude/skills/`, `plugins/*/skills/*/SKILL.md`, `plugins/*/docs/`, `meta/`

---

## 1. 見出し・タグ規約 [MANDATORY]

**なぜ**: タグを見出しに付けることで、AIが文書をスキャンする際に重要度を即座に判断できる。説明文中に埋もれると見落としのリスクが高まる。

### 1.1 タグの種類と意味

| タグ          | 意味               | 使用場面                               |
| ------------- | ------------------ | -------------------------------------- |
| `[MANDATORY]` | 必須遵守。例外なし | 絶対に従うべきルール、必ず読むべき文書 |
| `[CRITICAL]`  | 重要度最高         | 違反すると重大な問題が発生する項目     |
| `[IMPORTANT]` | 重要               | 注意が必要な項目                       |

**推奨**: `[MANDATORY]` を最優先で使用。AIへの効果が最も高い。

### 1.2 タグの配置ルール [MANDATORY]

**正しい例:**

```markdown
## 必須参照文書 [MANDATORY]
```

**誤った例:**

```markdown
## 必読文書

下記を読むこと [MANDATORY] ← タグは見出しに付けるべき

## 必読文書 [MANDATORY] ← スペースは1個のみ

## 必読文書 [MANDATORY] ← 末尾に余分なスペース
```

**ルール:**

1. タグは**見出し行の末尾**に付ける（説明文ではなく）
2. タグ前のスペースは**1個**
3. タグ後に余分なスペースを入れない

### 1.3 見出しレベルの使い分け

| レベル  | 用途                             | 例                            |
| ------- | -------------------------------- | ----------------------------- |
| `#`     | ファイルタイトル（1ファイル1個） | `# 設計書作成ワークフロー`    |
| `##`    | 主要セクション                   | `## 必須参照文書 [MANDATORY]` |
| `###`   | サブセクション                   | `### Domain層のタスク`        |
| `####`  | 詳細項目                         | `#### 確認 [MANDATORY]`       |
| `#####` | 使用しない                       | 階層が深すぎる                |

**原則**: 見出しは`####`（4レベル）まで。それ以上は構造を見直す。

## 2. 用語統一規約 [MANDATORY]

### 2.1 統一用語一覧

| 旧表現（非推奨）       | 新表現（推奨）                | 理由                                   |
| ---------------------- | ----------------------------- | -------------------------------------- |
| 必読文書               | 必須参照文書                  | 「必須」の方が強い印象                 |
| 共通必読文書           | 共通 - 必須参照文書           | ハイフン区切りで「必須参照文書」を統一 |
| レイヤー別必読文書     | レイヤー別 - 必須参照文書     | 同上                                   |
| ワークフロー別必読文書 | ワークフロー別 - 必須参照文書 | 同上                                   |

### 2.2 強調フレーズ（英語） [MANDATORY]

AIへの効果を最大化するため、重要な指示には英語フレーズを使用する。

| フレーズ          | 意味               | 使用場面                 |
| ----------------- | ------------------ | ------------------------ |
| `**NEVER skip.**` | 絶対にスキップ禁止 | 必須参照文書の説明の冒頭 |
| `**NEVER**`       | 絶対禁止           | 禁止事項の強調           |
| `**MUST**`        | 必須               | 必須事項の強調           |
| `**DO NOT**`      | してはいけない     | 禁止事項                 |
| `**ALWAYS**`      | 常に               | 常時遵守事項             |

**使用例:**

```markdown
## 必須参照文書 [MANDATORY]

**NEVER skip.** 下記を全て読み込み、深く理解すること

- `docs/rules/document_writing_rules.md`
```

### 2.3 長音ルール [MANDATORY]

原則として**JIS Z 8301（工業標準化）**に準拠し、語尾の長音を省略する。

| 正（長音省略） | 誤（長音あり） |
| -------------- | -------------- |
| ディレクトリ   | ディレクトリー |
| カテゴリ       | カテゴリー     |
| ファクトリ     | ファクトリー   |
| エントリ       | エントリー     |
| メモリ         | メモリー       |
| プロパティ     | プロパティー   |
| リポジトリ     | リポジトリー   |

**例外**: 固有名詞やフレームワーク公式名称はそのまま使用する。

### 2.4 説明文パターン

**必須参照文書セクションの説明文:**

```markdown
**NEVER skip.** 下記を全て読み込み、深く理解すること
```

**禁止事項の説明文:**

```markdown
**NEVER** 以下の操作を行ってはいけない
```

## 3. 文書構成規約

**なぜ**: 文書タイプごとに構成を統一することで、AIと人間の両方が「どこに何が書いてあるか」を予測できる。これにより読解速度と精度が向上する。

### 3.1 必須セクション（文書タイプ別）

#### ワークフロー文書

```markdown
---
name: workflow_name
description: 説明
applicable_when:
  - 適用条件1
  - 適用条件2
---

# タイトル [MANDATORY]

## 必須参照文書 [MANDATORY]

**NEVER skip.** 下記を全て読み込み、深く理解すること

- 参照文書リスト

## 概要

（このワークフローの目的と概要）

## 実行手順

### Step 1: ...

### Step 2: ...
```

#### ルール文書（`docs/rules/`）

```markdown
# タイトル

## 基本原則

（このルールの基本的な考え方）

## 詳細ルール

### サブセクション [MANDATORY]

（具体的なルール）

## 例外

（例外がある場合のみ）
```

### 3.2 メタデータ（frontmatter）

SKILL 文書（`plugins/*/skills/*/SKILL.md`、`.claude/skills/*/SKILL.md`）およびワークフロー文書は frontmatter を持つ。

```yaml
---
name: 識別名（必須）
description: 説明（必須）
user-invocable: true/false（SKILL のみ）
allowed-tools: ツール名カンマ区切り（SKILL のみ、任意）
argument-hint: 引数の説明（SKILL のみ、任意）
applicable_when:（ワークフローのみ、任意）
  - 適用条件 1
  - 適用条件 2
---
```

**ルール:**

- frontmatter がある文書では `name` と `description` は必須

## 4. 強調・フォーマット規約

### 4.1 強調の使い方

| マークアップ   | 用途                         | 例                                           |
| -------------- | ---------------------------- | -------------------------------------------- |
| `**太字**`     | 重要語句、キーワード         | `**NEVER skip.**`                            |
| `*斜体*`       | 使用しない                   | 日本語に不向き                               |
| `` `コード` `` | ファイル名、コマンド、変数名 | `` `docs/rules/document_writing_rules.md` `` |
| `> 引用`       | 注意事項、補足               | 控えめに使用                                 |

### 4.2 コードブロック

````markdown
# 言語指定あり（推奨）

```python
def foo(): ...
```
````

# 言語指定なし（構造説明など）

```
docs/
├── rules/
└── specs/
```

# インラインコード

`ファイル名` または `変数名`

````
### 4.3 リストの使い方

```markdown
# 順序なしリスト（複数項目の列挙）
- 項目1
- 項目2
- 項目3

# 順序ありリスト（手順・優先順位）
1. Step 1
2. Step 2
3. Step 3

# 入れ子（2段階まで）
- 親項目
  - 子項目1
  - 子項目2
````

**原則**: 入れ子は**2段階まで**。それ以上は構造を見直す。

### 4.4 表の使い方

```markdown
| ヘッダー1 | ヘッダー2 | ヘッダー3 |
| --------- | --------- | --------- |
| セル1     | セル2     | セル3     |
| セル4     | セル5     | セル6     |
```

**原則**:

- 列数は**5列以内**
- 複雑な情報は表ではなくリストで記述

### 4.5 文書参照の記法 [MANDATORY]

#### specs/ 内の文書を参照する場合

ID ベース参照記法は `plugins/forge/docs/spec_format.md` の「ID 参照記法」セクションを参照する。

#### `docs/rules/`、`.claude/`、`plugins/` の文書を参照する場合

**絶対パス**（プロジェクトルート起点）を使用。

```markdown
詳細は `plugins/forge/docs/spec_format.md` を参照
```

**理由**:

- ID 体系がないため、パスで参照する必要がある
- 相対パスは参照元ファイル移動時に壊れる

#### パス表記ルール [MANDATORY]

**基本原則**: パス参照は**フルパス**（プロジェクトルート起点）で記述する。

**省略が許容される条件**:

1. **同一段落内**または**同一リストアイテム内**で、直前にフルパスが出現している
2. 省略形は**ファイル名のみ**または**末尾ディレクトリ名のみ**
3. 文脈から一意に特定できる場合のみ

**省略できないケース**:

- 表（テーブル）内
- コードブロック（``` ）内
- 見出し内
- 別段落・別リストアイテム
- 中間パスの省略（`docs/rules/.../file.md` 形式）

**例:**

```markdown
# OK: 同一段落内で省略

`docs/rules/document_writing_rules.md` を読んでから `document_writing_rules.md` に従って実行する。
```

## 5. ドキュメントの性質

**なぜ分離するか**: 同じ情報を複数の場所に書くと、保守のたびに矛盾が生まれる。役割を明確に分離することで、AIと人間の判断コストが下がり、文書の再利用性が向上する。

| ディレクトリ        | 性質                                 | 説明                                         |
| ------------------- | ------------------------------------ | -------------------------------------------- |
| `docs/rules/`       | プロジェクトルール                   | 開発ルール・実装ガイドライン・文書記述ルール |
| `docs/specs/`       | プロジェクト固有仕様                 | 要件定義書・設計書・計画書                   |
| `plugins/*/docs/`   | プラグイン内部仕様                   | プラグインが配布する原則・フォーマット       |
| `.claude/skills/`   | プロジェクトローカル SKILL（非配布） | bw-cc-plugins 内部運用専用の SKILL           |
| `plugins/*/skills/` | 配布 SKILL                           | マーケットプレイス経由で配布される SKILL     |

**`docs/rules/` 内の役割分担:**

| 種別     | 役割                       | 例                                                      |
| -------- | -------------------------- | ------------------------------------------------------- |
| Rules    | どう書くか・どう実装するか | `document_writing_rules.md`, `cli_output_formatting.md` |
| Workflow | 何をするか（手順と順序）   | （必要に応じて追加）                                    |
| Notes    | 個別の留意点               | `skill_authoring_notes.md`                              |

## 6. `docs/rules/` 内の抽象例・命名規則 [MANDATORY]

`docs/rules/` 配下の文書は汎用ルールを記述する。
そのため、文書中の例文・プレースホルダーにプロジェクト固有の名称を使わない。

実装コードそのものの命名規則（型名・変数名・ファイル名等）は本文書の対象外とし、
各プロジェクトのコーディング規約に従う。

### 6.1 基本原則

- **文書中の例示**: プロジェクト固有の固有名を使わず、`Foo` / `Bar` 等のプレースホルダーを使用
- **日本語説明文**: 汎用的な業務用語（「アイテム」「カテゴリ」等）
- **業務シナリオ例**: 抽象的なデータ管理シナリオを使用

### 6.2 日本語説明文の置換規則

| プロジェクト固有（❌）       | 汎用表現（✅）           | 用途                               |
| ---------------------------- | ------------------------ | ---------------------------------- |
| 連絡先                       | アイテム、データ         | 主要なデータ単位                   |
| グループ                     | カテゴリ                 | データの分類                       |
| 連絡先管理アプリ、Contact[B] | {アプリ名}               | アプリ全体（プレースホルダー表記） |
| 電話番号、メールアドレス     | フィールドA、フィールドB | データ属性                         |
| 発信、メール送信             | アクションA、アクションB | 操作                               |
| 連絡先リスト画面             | 一覧画面、リスト画面     | 画面名                             |

### 6.3 プレースホルダー名

汎用的なプレースホルダーとして以下を使用する:

| プレースホルダー | 主な用途                |
| ---------------- | ----------------------- |
| `Foo`            | 主要な例（第 1 のもの） |
| `Bar`            | 第 2 の例               |
| `Hoge`           | 第 3 の例（必要時のみ） |
| `Fuga`           | 第 4 の例（必要時のみ） |

### 6.4 日本語説明文の例

**正しい例**:

```markdown
## データ取得フロー

Repository が API からアイテムを取得し、Service でビジネスロジックを適用後、
DataStore に保存して UI に反映する。
```

**誤った例**:

```markdown
## 連絡先データのフロー ← ❌ プロジェクト固有

Repository（ContactRepository）が連絡先を取得し... ← ❌
```

固有名詞（OS 公式フレームワーク名、標準ライブラリ名等）は文書中で言及する必要がある場合に限り使用してよい。

## 7. specs/ 内の文書記述規則 [MANDATORY]

### 7.1 メタデータ「関連要件」の記載ルール

#### 記載すべき内容

- 文書本文中で参照している全てのID（BL-006、SCR-001等）
- この文書を参照している他文書のID（双方向で記載）

#### 更新タイミング

- 文書作成時: 本文で参照するIDを全て記載
- 文書更新時: 新しいIDを参照したらメタデータにも追加

## 9. よくある間違いと修正例

### 9.1 タグの配置ミス

```markdown
# 誤り

## 必読文書

下記を全て読み込み、深く理解すること [MANDATORY]

# 正しい

## 必須参照文書 [MANDATORY]

**NEVER skip.** 下記を全て読み込み、深く理解すること
```

### 9.2 スペースの揺れ

```markdown
# 誤り（スペース2個）

## 必読文書 [MANDATORY]

# 誤り（末尾スペース）

## 必読文書 [MANDATORY]

# 正しい

## 必須参照文書 [MANDATORY]
```

### 9.3 用語の揺れ

```markdown
# 誤り（旧表現）

## 必読文書

# 正しい

## 必須参照文書 [MANDATORY]
```

### 9.4 説明文の揺れ

```markdown
# 誤り（日本語のみ）

下記のドキュメントを全て読み込み、深く理解すること

# 正しい（英語フレーズで強調）

**NEVER skip.** 下記を全て読み込み、深く理解すること
```

## 10. 検証方法

### 10.1 手動検証

以下のGrepコマンドで揺れを検出:

```bash
# スペース2個の検出
grep -r "  \[MANDATORY\]" docs/rules/ docs/specs/ plugins/ .claude/skills/ meta/

# 旧表現「必読文書」の検出
grep -r "## 必読文書" docs/rules/ docs/specs/ plugins/ .claude/skills/ meta/

# 説明文の旧パターン検出
grep -r "下記のドキュメントを全て読み込み" docs/rules/ docs/specs/ plugins/ .claude/skills/ meta/
```

### 10.2 修正時のチェックリスト

- [ ] タグは見出し行に付いているか
- [ ] タグ前のスペースは1個か
- [ ] 「必読文書」→「必須参照文書」に統一されているか
- [ ] 説明文は `**NEVER skip.**` で始まっているか
