## コンテキスト

`/forge:start-plan` は設計書からタスクを抽出し計画書を作成するスキルである。
現状のフローは以下の通り:

1. コンテキスト収集（設計書・ルール取得）
2. 文書の読み込み
3. 設計書からタスクを抽出・分割
4. AI レビュー

Phase 2（文書読み込み）から Phase 3（タスク抽出）への遷移は、設計書のモジュール一覧・クラス図を機械的にタスクに分解するだけであり、**実装アプローチの戦略的判断**が欠落している。

---

## 問題

### 設計と計画の間の質的断絶

- **設計書** = 目標状態の記述（完成したシステムの構造）
- **タスク一覧** = 作業指示の羅列（何をどの単位でやるか）

この2つの間に「目標状態にどういう経路で辿り着くか」という**到達戦略**が存在しない。

### 「依存される側から先に」は戦略ではない

現状の唯一の順序決定ルールは局所的な制約に過ぎず、以下のような戦略的判断を含まない:

| 戦略                 | アプローチ                 | 適用場面                             |
| -------------------- | -------------------------- | ------------------------------------ |
| ボトムアップ         | 基盤→上位層の順            | 安定した基盤が必要な場合             |
| スケルトン先行       | stub で全層を通す→肉付け   | 早期 E2E 検証が重要な場合            |
| リスク駆動           | 不確実性の高い部分から着手 | 技術リスクが大きい場合               |
| フィーチャースライス | 最小機能セットを縦断実装   | 早期デモ・フィードバックが必要な場合 |

### 手戻りコスト

戦略なしでタスクを生成すると、実装途中で「本来はスケルトン先行で作るべきだった」と判明し、大規模な手戻りが発生する。タスク再生成コスト >> 事前に戦略を練るコスト。

---

## 決定

### start-plan の Phase 2 と Phase 3 の間に「実装戦略策定」フェーズを挿入する

新しいフロー:

1. コンテキスト収集
2. 文書の読み込み
3. **実装戦略の策定** ← 新規
4. タスク抽出・分割（戦略に基づく）
5. AI レビュー

### 実装戦略は独立した Markdown ファイルとして出力する

- ファイル名: `{feature}_strategy.md`
- 配置先: 計画書と同じディレクトリ（plan/）
- 計画書（plan.yaml）と同様、**実装完了後に削除する** ephemeral 文書

### 実装戦略に含める内容

| セクション     | 内容                                                   |
| -------------- | ------------------------------------------------------ |
| アプローチ選択 | 上記4戦略（または組み合わせ）から選択し、根拠を記述    |
| フェーズ分割   | 大枠の実装フェーズ（2〜4フェーズ程度）                 |
| 検証ポイント   | 各フェーズ完了時に検証すべき事項                       |
| リスクと対策   | 技術的リスクと、どのフェーズで早期に潰すかの方針       |
| 既存資産の活用 | 設計書の「使用する既存コンポーネント」をどう組み込むか |

### 実装戦略に含めない内容

- 個別タスクの詳細（それはタスク一覧の責務）
- 設計書の内容の転記（参照で十分）
- 定量的なスケジュール見積もり（AI が推測すべきでない）

---

## 根拠

### 保守コストの懸念は無効

計画文書（plan.yaml, strategy.md）は実装完了後に削除する ephemeral 文書である。設計書のように長期保守する必要がないため、別ファイル化による保守コスト増の懸念は当てはまらない。

### 設計書に含めるべきでない理由

設計書は「完成形の構造」を記述する文書であり、**時間軸を持たない**。「フェーズ1でここまで作り、フェーズ2でここを繋ぐ」という記述は本質的に時間的・順序的情報であり、設計書の What/How 境界ガイドの思想に照らして設計書にはそぐわない。

### plan.yaml に含めるべきでない理由

plan.yaml は構造化データ（YAML）であり、`start-implement` が機械的にパースする。フリーテキストの戦略記述を混在させると、データとしての扱いやすさが損なわれる。

### 変更が必要なファイル

| ファイル                                                         | 変更内容                                     |
| ---------------------------------------------------------------- | -------------------------------------------- |
| `plugins/forge/skills/start-plan/SKILL.md`                       | Phase 2.5（実装戦略策定）の追加              |

- `plan_format.md` — plan.yaml のスキーマは変更なし
- `start-implement` — タスク実行ロジックは影響なし（strategy.md は参考文書として required_reading に含めうる）
