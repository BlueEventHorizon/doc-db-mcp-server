> 対象: forge プラグイン全スキル

---

## 1. 背景

forge スキルは多くの作業工程を持つ（文書収集・分析・実行・確認・後処理）。
全てを単一のコンテキストで実行すると以下の問題が発生する（God-Skill 問題）:

1. **コンテキスト消費**: 長い SKILL.md + 参照文書 + 中間結果がコンテキストを圧迫する
2. **追従精度の低下**: 指示が長くなるほど AI が正確に追従しきれない
3. **並列処理の不可**: 独立した調査（仕様書検索・ルール検索・コード探索）を直列で実行せざるを得ない
4. **コンテキスト圧縮による消失**: 長時間セッションで中間結果が失われる

### 先行事例

review ワークフロー（`review/SKILL.md`）で実証済み:

- `review`（オーケストレータ）→ `reviewer` → `evaluator` → `fixer` の分業構造
- 単一スキルが全工程を担う旧設計（God-Skill）から移行し、品質・効率が向上した

---

## 2. 要件

### FNC-001: オーケストレータ設計

forge の user-invocable スキルはオーケストレータとして設計する。

- オーケストレータは全体の進行管理・判断分岐・ユーザー対話を担う
- 詳細な作業（文書検索・レビュー実行・修正等）は汎用 Agent / fork 型 SKILL に委譲する
- オーケストレータの SKILL.md には「何を委譲するか」を書き、「どう実行するか」は書かない

### FNC-002: セッションディレクトリ通信

オーケストレータと agent はセッションディレクトリ経由でデータを受け渡す。

- オーケストレータがセッションディレクトリを作成し、`session.yaml` を初期化する
- agent はセッションディレクトリのパスを受け取り、必要なファイルを読み書きする
- プロンプト経由のデータ受け渡しは最小限にする（パスのみ）

### FNC-003: 参照文書の自己完結性

agent に渡す参照文書（ガイド・フォーマット定義等）は自己完結型とする。

- agent がその文書だけ読めば作業を自律実行できること
- 前提知識・外部コンテキストへの依存を排除すること
- `plugins/forge/docs/` に配置する（プロジェクト固有文書にオーバーライドされない真実）
- **例外**: 特定スキル専用の観点定義（他オーケストレーターが直接参照しない文書）は、当該スキルの `docs/` ディレクトリに配置してよい

### FNC-004: 並列実行可能な構造

独立した調査工程は並列実行できる構造にする。

- 仕様書検索・ルール検索・PR検索・コード探索は互いに独立
- 各 agent が個別のファイルに結果を書き込み、ファイル競合を防ぐ
- オーケストレータが結果を統合して次の工程に渡す

### FNC-005: 文書フォーマットの選定基準

forge が生成・管理する文書のフォーマットは、文書の性質に応じて選定する。

- **YAML**: 機械的なパース・更新が必要な文書に使用する
  - 計画書（タスクID・優先度・依存関係・ステータス等、全て構造化データ）
  - `start-implement` がタスク選択・依存チェック・完了更新を確実に行うために構造化が必須
- **Markdown**: 自由記述・図表・コード例を含む文書に使用する
  - 要件定義書（ユーザーストーリー・ビジネスルール等の自然言語記述）
  - 設計書（mermaid フローチャート・アーキテクチャ図・シーケンス図・コード例）

YAML は機械処理の確実性を保証するが、mermaid 等の埋め込みに不向き。
Markdown は表現力が高いが、AI によるパースが不確実（テーブル列幅崩れ・`<br>` 改行等）。
文書の主要な消費者（機械 vs 人間）に応じて適切なフォーマットを選択する。

### FNC-006: 共通完了処理フロー

文書生成系オーケストレーター（start-requirements, start-design, start-plan）は成果物作成後に以下の完了処理を実行する。

1. `/forge:review {type} --files {差分ファイル} --auto` — AIレビュー+自動修正（対象はオーケストレーターが作成・変更したファイルのみ）
2. `/create-specs-toc` — ToC 更新（利用可能な場合のみ）
3. `/anvil:commit` — commit/push 確認

> **注記**: start-implement はコード実装オーケストレーターであり、ToC 更新は不要なため review → commit の2ステップで完了する。

- 独立した finalize スキルは設けない。完了処理は各オーケストレーターが直接実行する
- review は `--auto` モードで実行し、人間レビューは review ワークフロー内で処理される

### FNC-007: ユーザー入力の解釈は AI が行う

オーケストレーターが受け取るユーザー入力（コマンド引数等）の解釈は、スクリプトではなく AI が直接行う。

- ユーザー入力には自然言語が混在するため、リジッドなトークンパーサーでは対応できない
- 引数解析用のスクリプト（Python 等）を作成してはならない
- 不足・曖昧な引数は AskUserQuestion で補完する
- スクリプトは構造化データ（YAML/JSON）の処理に限定する
