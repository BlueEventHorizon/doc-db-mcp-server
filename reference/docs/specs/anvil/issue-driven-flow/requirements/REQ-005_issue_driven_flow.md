# REQ-005 issue-driven-flow 要件定義: GitHub Issue を起点にした SDD パイプライン連携

## メタデータ

| 項目       | 値                                                    |
| ---------- | ----------------------------------------------------- |
| 要件 ID    | REQ-005                                               |
| feature ID | issue-driven-flow                                     |
| 種別       | 要件定義（What のみ。手段・配置・実装詳細は含まない） |
| 作成日     | 2026-04-25                                            |
| 対象       | anvil プラグイン                                      |
| 関連要件   | COMMON-REQ-002（エラーハンドリング方針）              |

> **注記**: `COMMON-REQ-*` はプロジェクト共通要件（恒久的な横断仕様）を指す識別子である。本 feature 要件定義書の frontmatter における「この文書が正」の優先ルールの対象外とし、共通基盤仕様として本文書と並列に扱う。

---

## 1. 背景

GitHub Issue は現状「課題の記録場所」としてのみ機能しており、開発プロセスとの連携が弱い。要件定義書・設計書・計画書・実装コードはリポジトリ内の成果物として蓄積されるが、Issue とは別管理となっており、Issue から成果物へ辿ることも、成果物の進行状況を Issue から把握することもできない。

一方、プロジェクトには SDD（仕様駆動開発）パイプラインが既に整備されており、要件定義 → 設計 → 計画 → 実装 を一貫して回せる。しかし現状は「Issue を入口として SDD を回す」ワークフローが存在しない。

本 feature は、GitHub Issue を開発プロセスの入口として確立し、既存の SDD パイプラインと連携させることで、「Issue から始めて PR の作成まで一連の流れで辿れる」状態を実現する。

---

## 2. 目的

以下 3 点を同時に満たす仕組みを提供する:

1. Issue を開発の入口として登録できる（問題・背景・原因の記録）
2. Issue を指定することで、その内容に応じた SDD パイプラインを起動できる
3. Issue から開発状況（どの成果物がどこにあるか、進捗はどこか）を辿れる

---

## 3. スコープ

### 3.1 含める

- Issue を作成する操作（Issue 種別 = Bug Report / Feature Request の選択と、種別ごとの必須セクション記録）
- Issue を起点に SDD パイプラインを回すオーケストレーション
- Issue の進捗と成果物リンクを Issue に書き戻す操作
- 実行時に種別（UI Issue / 非 UI Issue 等）をユーザーが選ぶ仕組み
- UI Issue のフローで Figma から取得すべき情報の種類

### 3.2 含めない（別 feature で扱う）

| 非対象                                                           | 理由                                                            |
| ---------------------------------------------------------------- | --------------------------------------------------------------- |
| 既存の SDD 機能そのものの改修                                    | 本 feature は既存の SDD 機能を利用する側である                  |
| 既存のコミット・PR 作成機能の改修                                | 本 feature は既存機能を呼び出す側である                         |
| 既存のドキュメント検索機能の改修                                 | 間接的に利用するのみ                                            |
| Issue 以外（Jira / Linear 等）起点のワークフロー                 | スコープを GitHub Issue に限定                                  |
| 軽微 Issue ルート（SDD 全工程を省略する高速経路）                | 種別分類の運用知見を得てから別 feature で検討                   |
| UI Issue フローにおける複数画面（複数 Figma フレーム）の取り扱い | 1 Issue は単一画面を対象とする。複数画面は Issue を分割すること |

## 5. 要件一覧

各要件には優先度（1〜99 の数値）を付与する。基準は以下のとおり:

- 70〜99 高: コア経路・不変条件・ブロッカー解消

### 5.1 FNC-01: Issue 作成操作

**優先度**: 85（コア経路: Issue 起票の入口）

**目的**: 問題または要望を、後続の SDD パイプラインが参照するに足る情報量で GitHub Issue として記録する。

**ユーザー視点の振る舞い**:

- ユーザーが問題または要望を記述すると、タイトルと本文が整理された Issue が GitHub に登録される
- Issue 種別は次の 2 種を扱う。種別ごとに本文の必須セクションが異なる:
  - **Bug Report**（既存挙動の不具合報告）
  - **Feature Request**（新機能・既存機能の改善要望）
- タイトルは次の条件を満たす:
  - 改行を含まない 1 文で記述される
  - Issue 種別が読み取れる（例: 「[Bug] / [Feature]」相当の識別が冒頭で可能であること。具体的な記号・前置詞は設計に委ねる）
  - 問題の対象と要点が読み取れる
  - GitHub Issue タイトルの上限（256 文字）以内である

**Bug Report の必須セクション**:

| セクション          | 内容                                                                                                                             |
| ------------------- | -------------------------------------------------------------------------------------------------------------------------------- |
| 背景 / コンテキスト | 何をしようとしていたか、いつ気付いたか                                                                                           |
| 現象（実際の動作）  | 実際に起きたこと。エラーメッセージがあれば原文を含める                                                                           |
| 期待動作            | 本来どう動作すべきだったか                                                                                                       |
| 再現手順            | 既知の初期状態（ログイン状態・データ準備等）を明記した上での、番号付きの操作列                                                   |
| 環境情報            | 少なくとも OS / バージョン / 使用ツールやランタイムのバージョン                                                                  |
| 重大度              | 影響度の自己申告（区分の名称・段階数は設計に委ねる。少なくとも「業務継続不能」「機能に支障あり」「軽微」が識別できる粒度を持つ） |
| 証跡                | 関連するスクリーンショット / ログ / 動画への参照（添付または外部リンク。存在しない場合は「なし」と明記する）                     |
| 原因                | 判明している場合のみ記載。不明な場合は「調査中」と明記する                                                                       |

**Feature Request の必須セクション**:

| セクション          | 内容                                                                                                                  |
| ------------------- | --------------------------------------------------------------------------------------------------------------------- |
| 背景 / コンテキスト | なぜこの要望が生まれたか、現状の課題は何か                                                                            |
| ユーザーストーリー  | 「〜（役割）として、〜（やりたいこと）したい。なぜなら〜（得たい価値）」の形式で、役割・行為・価値の 3 要素を明示する |
| 受け入れ基準        | 本要望が満たされたと判断できる客観的な条件。各条件は単独で真偽判定できる粒度で記述する                                |
| 代替案 / 補足       | 検討した代替案または補足情報。存在しない場合は「なし」と明記する                                                      |

**不変条件**:

- 必須セクションのうち情報が得られない項目は、空欄ではなく閉じた集合の不在表現「不明」「なし」「調査中」「該当なし」のいずれかで埋める（読み手が「未記入なのか不在なのか」を判別できないことを防ぐ。「N/A」「未確認」「要調査」等の別表記は許容しない）
- 種別ごとの必須セクションは省略しない。情報が不足する場合でも、不在表現でセクション自体は残す
- 必須セクションの見出し名は本要件で定めた正規名で固定する。別名・英訳・略称は許容しない（例: 「背景 / コンテキスト」を「Background」「背景」「コンテキスト」単独で代替することは不可）。これは FNC-03 / I-03 の保護対象セクションを before/after 比較で機械的に識別可能にするため
- Issue 種別ごとに、以下の中核セクションは不在表現のみで埋めてはならない（最低情報密度）。中核セクションがすべて不在表現の場合、FNC-01 の要件を満たさず、FNC-02 を開始しない:
  - Bug Report: 「現象（実際の動作）」「再現手順」
  - Feature Request: 「ユーザーストーリー」「受け入れ基準」
- Feature Request のうち UI に関わる要望（画面・レイアウト・視覚表現の追加・変更を含むもの）の場合、本文内に Figma 参照情報（Figma URL / 画面名 / 対象フレームの識別子のいずれか 1 つ以上）を記載する。これは UI Issue 経路（FNC-04 で UI Issue が選択される可能性がある Issue）で FNC-05 / ERR-01 が参照する入力を FNC-01 段階で確保するため
- 解決内容（対策・実装計画・成果物リンク）は本操作の時点では記録しない。FNC-03（書き戻し）で後から追記する

**含まないこと**:

- 種別の自動判定（タイトル文字列・添付物の有無等からの推定）は行わない。種別は Issue 作成時にユーザーが選択する
- 重大度・受け入れ基準等の自動生成は行わない。ユーザー入力を整形するに留める

### 5.2 FNC-02: Issue 起点フローのオーケストレーション

**優先度**: 90（コア経路: SDD 起動）

**目的**: Issue 起点フローを構成する SDD パイプラインと後工程の PR 作成を、ユーザーから見て一連の流れとして実行する。

**ユーザー視点の振る舞い**:

- ユーザーが Issue 番号を指定すると、以下が順に進行する:
  1. Issue の内容（タイトル・本文・ラベル、および FNC-01 で記録された種別別必須セクション）が読み取られる。FNC-01 の種別別必須セクションは後続 SDD 工程（要件定義 → 設計 → 計画 → 実装）の入力として参照される
  2. 種別が対話的に確認される（FNC-04）
  3. 選ばれた種別に応じた SDD 工程（要件定義 → 設計 → 計画 → 実装）が順に進行する。各工程は FNC-01 の種別別必須セクションを入力として参照し、情報が不足する場合は当該工程でユーザーに追加確認する。UI Issue の場合は FNC-05 の Figma 由来情報取得を伴う
  4. 各工程の完了時点で、成果物へのリンクと進捗が Issue に書き戻される（FNC-03）
  5. SDD パイプライン完了後、PR 作成工程に遷移する（遷移トリガーの具体形は TBD-005）
  6. PR が作成される（PR 作成工程の完了判定基準は TBD-004）
- PR と Issue を結びつけ、PR マージ時に Issue が自動クローズされる参照を PR に含めるかは TBD-006 で確定する

**種別別のフェーズ経路**:

| 種別        | 要件定義 | 設計                                      | 計画 | 実装 |
| ----------- | -------- | ----------------------------------------- | ---- | ---- |
| UI Issue    | 実施     | UI 設計（FNC-05 の Figma 由来情報を含む） | 実施 | 実施 |
| 非 UI Issue | 実施     | 一般設計                                  | 実施 | 実施 |

個別の実装手段（各工程と実装単位の対応関係等）は設計フェーズに委ねる。

**各工程の「完了」判定**:

- 各工程は次をすべて満たした時点で「完了」とみなす:
  - 当該工程の成果物ファイルが存在する
  - 成果物がリモートリポジトリに push 済みである（未 push は完了扱いしない。FNC-03 不変条件との整合）

**異常系**:

- SDD パイプライン実行中に工程が失敗した場合の振る舞いは TBD-003 で確定する（候補: 失敗工程を Issue に書き戻す、以降の工程を自動起動せず中断する、ユーザーに再開判断を求める、等）。詳細は COMMON-REQ-002（エラーハンドリング方針）に従う。
- PR 作成工程が失敗した場合の振る舞い（Issue への失敗ステータス書き戻し要否、手動再実行の案内等）は TBD-003 で確定する。

**含まないこと**:

- 既存の SDD 機能そのものの改修。本要件は既存機能を順次呼び出すのみであり、既存機能の内部には手を入れない

### 5.3 FNC-03: Issue 書き戻し操作

**優先度**: 85（コア経路: Issue 書き戻し）

**目的**: Issue 本文に成果物へのリンクと進捗ステータスを反映する。

**ユーザー視点の振る舞い**:

- FNC-02 の進行に応じて、Issue 本文に以下が書き戻される:
  - 各成果物（要件定義書・設計書・計画書・実装コード）への参照リンク。PR は SDD パイプラインの後工程として既存の PR 作成機能の責務で作成されるため、PR URL は本書き戻し対象外とし、FNC-02 の PR 作成工程でユーザーに通知する形で扱う
  - 進捗ステータス。進捗ステータスは SDD パイプライン内の工程のみを対象とし、少なくとも次を識別できる:
    - 現在進行中の工程（要件定義 / 設計 / 計画 / 実装）
    - 完了済みの工程
    - PR 作成工程は SDD パイプライン外の後工程であり、本書き戻しの進捗ステータスには含めない（既存の PR 作成機能の責務として分離）
    - 具体的な表現形式（マーカー形式・テンプレート文字列等）は TBD-002 で設計に委ねる

**不変条件**:

- FNC-01 で記録されたユーザー記述セクション（Bug Report の「背景 / コンテキスト」「現象」「期待動作」「再現手順」「環境情報」「重大度」「証跡」「原因」、Feature Request の「背景 / コンテキスト」「ユーザーストーリー」「受け入れ基準」「代替案 / 補足」）は、書き戻しで**上書きしない**。これらは FNC-01 で記録された情報を保持する
- ユーザー記述セクションは書き戻し前後で本文が一字一句不変であること。書き戻し操作は、ユーザー記述セクションと明確に区別された専用セクション（成果物リンクセクション・進捗ステータスセクション等）にのみ行い、ユーザー記述セクション内への追記・変更・末尾追記を含めて一切行わない。これは I-03 の保護を before/after 比較で機械的に検証可能にするため
- 成果物の本文を Issue にコピーしない（NFR-02 を参照）
- 書き戻しは冪等である。同一工程の再実行時は既存の書き戻し内容を追記せず上書きする（Issue 本文が肥大化しないようにする）
- 書き戻し操作が失敗した場合の振る舞いは COMMON-REQ-002（エラーハンドリング方針）に従い、静かに成功扱いしない
- 参照リンクは、次の静的基準に該当するもののみ記載する:
  - ブラウザで閲覧可能な永続的な URL であること
  - リポジトリ内の成果物の場合、リモートに push 済みであること（未 push のファイルは記載しない）
  - Figma デザインの場合、Figma 上で当該フレームを参照できる永続的な URL であること
- 権限起因の閲覧可否（プライベートリポジトリでの閲覧権限、Figma の共有設定等）は書き戻し側の責務外とする
- リポジトリ内の成果物に対する書き戻しは、当該成果物が push 済みであることを前提とする。push が完了していない時点で書き戻しを実行してはならない（順序: 成果物作成 → push → 書き戻し）
- push が失敗した場合、当該成果物に対する書き戻しは実行しない。失敗時の振る舞いは COMMON-REQ-002（エラーハンドリング方針）および TBD-003 に従う

### 5.4 FNC-04: 種別選択（実行時）

**優先度**: 70（分岐制御: コア経路の一部）

**目的**: Issue の性質に応じて SDD パイプラインの経路を分岐する。

**ユーザー視点の振る舞い**:

- FNC-02 の実行時、ユーザーに種別を対話的に確認する
- 最小実装として次の 2 種を必須とする（「Figma 由来情報を参照するか否か」の軸は必須）:
  - UI Issue（Figma 由来情報を伴う）
  - 非 UI Issue（Figma 由来情報を伴わない）
- 追加の粒度（3 種 / 4 種等の細分化軸）の導入可否は設計・実装時に決定する（TBD-001）
- 選択された種別に応じた経路で以後の SDD パイプラインが実行される

**不変条件**:

- 種別の自動判定ロジック（Issue 本文・ラベルからの推定ルール等）は本要件に含めない。誤判定リスクを避け、将来の粒度変更に柔軟性を残すため、**常にユーザーに確認する**

---

### 5.5 FNC-05: UI Issue における Figma 由来情報

**優先度**: 60（UI Issue 経路のみに影響）

**目的**: UI Issue のフローで、Figma デザインから SDD パイプラインに必要な情報を取り込む。

**ユーザー視点の振る舞い**:

- UI Issue が選択された場合、Issue に記載された Figma 情報（Figma URL / 画面名 / 対象フレームの識別子のいずれか、または複数）を入力として、以下の情報が取得されて UI 設計フェーズで参照可能になる:
  - 対象フレームの Figma 上の識別子（Figma 上で当該フレームを一意に参照するための識別子）と永続的な URL
  - デザイン詳細（少なくとも: レイアウト / 要素構成 / スタイル）
  - デザイントークン（少なくとも: 色 / 余白 / 書体）
  - スクリーンショット
- 本 feature では 1 Issue につき単一の対象フレームのみを扱う（複数画面が対象となる場合は Issue を分割する。§3.2 参照）

**不変条件**:

- Issue に記載された Figma URL / 画面名 / 識別子（Figma 上のフレーム識別子）は、いずれの形式であっても信頼せず、Figma 側で対象フレームを確認して正しい識別子を確定する（古い / 誤った情報によるズレを防ぐ）
- 複数の情報が指定されかつ矛盾する場合は、Figma 側の存在確認結果を優先する

**異常系**:

- Figma 由来情報の取得が失敗した場合（Figma API エラー、対象フレームの識別子が Figma 上で確認できない、スクリーンショット生成失敗等）の振る舞いは TBD-003 で確定する。候補: UI Issue フローを中断しユーザーに手動指定を求める / 別フレームの再探索を促す / 非 UI Issue に切り替える。詳細は COMMON-REQ-002（エラーハンドリング方針）に従う。
- 識別子が複数一致した場合: 候補一覧を提示し、ユーザーに選択を求める

---

### 5.6 NFR-01: 依存方向の制約

**優先度**: 90（不変条件: 依存方向）

- 本 feature の全 skill は anvil プラグイン配下に配置する
- 本 feature は forge の既存 skill を呼び出してよいが、forge の実装には一切手を入れない
- forge 側から anvil に依存してはならない（anvil → forge の一方向依存）
- 本 feature 実装中に forge の改修が必要と判明した場合、本 feature 内で forge を改修してはならない。別 Issue を起票し、独立した feature として要件定義からやり直す

### 5.7 NFR-02: Issue 索引性

**優先度**: 90（不変条件: Issue 索引性）

- Issue は成果物への「索引」として機能する
- 成果物の本文（要件定義書・設計書・計画書・実装コード）を Issue 本文に複製しない
- 成果物の正本はリポジトリ内の所定の場所に保存されたファイル、および GitHub 上のコード本体である（具体的な配置先は設計フェーズで規定する）
- Issue 本文に記載するのは、成果物への**参照（リンク）**と**進捗ステータス**のみである

### 5.8 NFR-03: 外部 API 制約と監査可能性

**優先度**: 50（外部依存の品質特性）

- 外部 API（GitHub / Figma）のレート制限に抵触した場合の待機・再試行方針は COMMON-REQ-002（エラーハンドリング方針）に従う
- 外部 API を呼び出す全操作（Issue 作成 / Issue 書き戻し / PR 作成 / Figma 由来情報取得）はローカルログに残し、事後に監査できるようにする。ログには少なくとも操作種別・対象 ID・実行時刻・成否を含める
- 本 feature 全体（FNC-02 のオーケストレーション）は SDD 全工程を内包する長時間処理であるため、応答時間目標は意図的にスコープ外とし、本要件では定義しない
- FNC-01 / FNC-03 等の個別操作（外部 API 1 回呼び出しで完結する操作）に対して応答時間目標を設けるか否かは TBD-007 で確定する

### 5.9 ERR-01: 前提条件未充足時の振る舞い

**優先度**: 90（不変条件: 全 FNC が依存する基盤要件。前提条件不充足の検出は横断的振る舞い）

以下の前提条件のいずれかが未充足の場合、FNC-01 / FNC-02 / FNC-03 は静かに失敗せず、**明示的にエラー終了し充足方法を案内する**:

| 前提条件                                                                                                                       | 対象              |
| ------------------------------------------------------------------------------------------------------------------------------ | ----------------- |
| gh CLI がインストールされ認証されていること                                                                                    | 全操作            |
| カレントディレクトリが git リポジトリ内であること                                                                              | 全操作            |
| リモートリポジトリが設定されていること                                                                                         | 全操作            |
| FIGMA_PAT 環境変数が設定されていること                                                                                         | UI Issue 経路のみ |
| Figma ファイル情報が参照可能であること（FIGMA_PAT を用いて Issue に記載された Figma URL のファイルメタ情報取得が成功すること） | UI Issue 経路のみ |

**受け入れ条件（本要件固有）**:

- 前提条件が未充足の場合、非ゼロの終了コードで処理を終了する
- エラーメッセージには、未充足の前提条件の名称と充足手順（または参照ドキュメント）を含める
- ERR-01 起因のエラーメッセージは stderr のみに出力し、標準出力には書き出さない

なお、責務境界として、ERR-01 の「Figma ファイル情報参照可能」はファイルレベル到達性（ファイル全体にアクセス可能か）を検証するものであり、FNC-05 の「識別子存在確認」（個別フレームの妥当性検証）とは別レイヤの検証である。

上記以外のエラー時の振る舞いは COMMON-REQ-002（エラーハンドリング方針）に従う。

## 6. 主要シナリオ

ユーザー視点のユースケース記述（実装コンポーネント間のシーケンス詳細は設計書に委ねる）:

**S1. Issue 起票（FNC-01）**

1. ユーザーが Issue 種別（Bug Report / Feature Request）を選択する
2. 種別に応じた必須セクション（Bug Report の場合: 「背景 / コンテキスト」「現象（実際の動作）」「期待動作」「再現手順」「環境情報」「重大度」「証跡」「原因」 / Feature Request の場合: 「背景 / コンテキスト」「ユーザーストーリー」「受け入れ基準」「代替案 / 補足」）の入力をユーザーが整理する
3. GitHub Issue が起票され、Issue 番号 `#N` が割り当てられる
4. Issue 本文に種別ごとの必須セクションが記録された状態になる（情報が得られない項目は「不明」「なし」「調査中」「該当なし」のいずれかで埋まっており、中核セクション（Bug Report は「現象（実際の動作）」「再現手順」、Feature Request は「ユーザーストーリー」「受け入れ基準」）は不在表現のみで埋まっていない）

**S2. Issue 起点の SDD 実行（FNC-02）**

1. ユーザーが Issue 番号を指定して開発を開始する
2. 種別（UI Issue / 非 UI Issue）が対話的に確認される（FNC-04）
3. 選択された種別に応じた SDD 工程（要件定義 → 設計 → 計画 → 実装）が順に進行する。UI Issue の場合は設計工程で Figma 由来情報の取得を伴う（FNC-05）
4. 各工程の完了時点で、成果物へのリンクと進捗ステータスが Issue に書き戻される（FNC-03）
5. SDD パイプライン完了後、PR 作成工程に遷移する（遷移トリガーの具体形は TBD-005）
6. PR が作成される。PR 作成工程の完了判定基準は TBD-004
7. PR と Issue を結びつけ、PR マージ時に Issue が自動クローズされる参照を PR に含めるかは TBD-006
8. PR がマージされると、紐づく Issue は自動的にクローズされる（TBD-006 の決定に依存）
