# セマンティック検索 技術調査レポート

本文書は semantic-query 要件定義の一部として、設計方針の判断材料となる技術調査の結果をまとめたものである。

調査日: 2026-03-29

---

## 1. 調査の背景

doc-advisor は現在、文書検索を以下の手順で行っている:

1. YAML ファイル（ToC）に全文書のメタデータを保存（850KB）
2. 検索時に AI が ToC を全量読み込む
3. AI が頭の中でキーワードを照合し、関連文書を選ぶ

この方式では、検索のたびに大量のコンテキスト（AI の作業メモリ）を消費する。
本調査は「検索を AI の外で実行する」ための技術選択肢を整理する。

---

## 2. 技術用語の解説

### 2.1 全文検索とインデックス

**全文検索**とは、文書の中身を対象にキーワードで検索する技術である。
Google 検索も全文検索の一種だが、ここでは手元のファイルを対象にした小規模な検索を指す。

**インデックス**とは、検索を高速にするための「索引」である。
本の巻末にある「索引」と同じ発想で、「この単語はどの文書に出てくるか」を事前に記録しておく。
索引があれば、全文書を毎回読み直す必要がなくなる。

### 2.2 BM25

**BM25**（Best Match 25）は、全文検索で最も広く使われるスコアリング手法である。

仕組みを簡単に言うと:

- 検索キーワードが文書に含まれるほど高スコア
- ただし、どの文書にも出てくるありふれた単語（「の」「する」等）は低評価
- 長い文書より短い文書で同じ回数出現する方が高評価

Google 検索の初期も BM25 に近い仕組みを使っていた。
40 年以上の歴史があり、シンプルだが強力で、現在でも多くのシステムの基盤になっている。

### 2.3 Embedding（埋め込み）とベクトル検索

**Embedding** とは、テキストを数百〜数千個の数値の列（ベクトル）に変換する技術である。

例えば「ログイン」というテキストを [0.12, -0.45, 0.78, ...] のような数値列に変換する。
重要なのは、**意味が近いテキスト同士は、ベクトルも近くなる**という性質である。

- 「ログイン」と「認証」→ ベクトルが近い（意味が似ている）
- 「ログイン」と「商品一覧」→ ベクトルが遠い（意味が異なる）

**ベクトル検索**とは、このベクトルの「近さ」で関連文書を見つける手法である。
キーワードが一致しなくても、意味が近ければ見つかるのが最大の利点である。

### 2.4 トークナイザー

**トークナイザー**とは、テキストを検索可能な単位（トークン）に分割する処理である。

英語は空白で単語が区切られるため、分割は容易である:

- "version management" → ["version", "management"]

日本語は空白がないため、分割が難しい:

- "バージョン管理" → どこで切る？

日本語の分割方法には以下がある:

| 方法                   | 分割例                                           | 特徴                                            |
| ---------------------- | ------------------------------------------------ | ----------------------------------------------- |
| 形態素解析（MeCab 等） | 「バージョン」「管理」                           | 高精度だが外部ツール必要                        |
| N-gram（bigram）       | 「バー」「ージ」「ジョ」「ョン」「ン管」「管理」 | 外部ツール不要だが冗長                          |
| 文字種境界             | 「バージョン」「管理」                           | カタカナ/漢字の切り替わりで分割。簡易だが実用的 |

### 2.5 ハイブリッド検索

**ハイブリッド検索**とは、BM25（キーワード一致）とベクトル検索（意味の近さ）を組み合わせる手法である。

- BM25 の強み: 「migration」で検索すれば「migration」を含む文書を確実に見つける
- BM25 の弱み: 「移行」で検索しても「migration」を含む文書は見つからない
- ベクトル検索の強み: 「移行」で検索しても「migration」の文書が見つかる（意味が近い）
- ベクトル検索の弱み: 完全一致のキーワードを見落とすことがある

両方を使うことで、互いの弱点を補完できる。

### 2.6 RRF と Convex Combination

2 つの検索結果を「1 つのランキング」に統合する手法:

- **RRF**（Reciprocal Rank Fusion）: 各検索結果の「順位」だけを見て統合する。スコアの大きさは無視する。チューニング不要で安定
- **Convex Combination**: 各検索結果の「スコア」を重み付きで足し合わせる。チューニングすると RRF より高精度

### 2.7 リランキング

**リランキング**とは、検索結果をさらに精密に並べ替える処理である。

1. まず BM25 やベクトル検索で候補を広く取得（高速だが粗い）
2. 次にリランカーが候補だけを精密に評価し直す（遅いが正確）

600 件程度の小規模コーパスでは、検索段階で十分な精度が得られるため、リランキングは通常不要である。

## 3. 調査結果

### 3.1 SQLite FTS5（全文検索エンジン）

**SQLite** は Python に標準搭載されているデータベースである。
**FTS5** はその全文検索拡張機能で、追加インストールなしで使える。

| 項目                              | 結論                                                          |
| --------------------------------- | ------------------------------------------------------------- |
| Python 標準ライブラリで利用可能か | **可**（macOS / Linux とも確認済み）                          |
| BM25 スコアリング                 | 対応。カラムごとの重み付けも可能                              |
| 日本語対応                        | trigram トークナイザーで 3 文字以上は検索可。2 文字以下は不可 |
| 日英混在テキスト                  | trigram なら英語も日本語もカバーできる                        |
| 性能                              | 600 件程度では問題にならない                                  |

**重要な発見**: doc-advisor のメタデータ（title, keywords, applicable_tasks）は英語で記述されているため、**日本語トークナイザーは当面不要**。英語のキーワード検索なら、デフォルトの空白分割（unicode61）で十分に機能する。

### 3.2 軽量検索ライブラリ

Python で使える全文検索ライブラリの調査結果:

| ライブラリ      | メンテ状態 | 外部依存  | 日本語 | 永続化 | 増分更新 |
| --------------- | ---------- | --------- | ------ | ------ | -------- |
| Whoosh-Reloaded | 低頻度     | なし      | 要追加 | あり   | あり     |
| tantivy-py      | 活発       | Rust 拡張 | 未対応 | あり   | あり     |
| bm25s           | 活発       | Numpy     | 要追加 | あり   | なし     |
| rank_bm25       | 停止       | なし      | 自前   | なし   | なし     |

**重要な発見**: BM25 のアルゴリズム自体は約 100 行の Python コードで実装できる。
外部ライブラリを使わず、標準ライブラリのみで BM25 スコアリングを自前実装する選択肢がある。

### 3.3 Embedding モデル + Vector DB

#### Embedding モデル

| モデル                        | 形態 | コスト          | 日英対応 | ローカル実行     |
| ----------------------------- | ---- | --------------- | -------- | ---------------- |
| OpenAI text-embedding-3-small | API  | $0.02/1M tokens | 良好     | 不可             |
| voyage-4-nano                 | OSS  | 無料            | 良好     | **可**（ollama） |
| Cohere embed-v4               | API  | $0.12/1M tokens | 優秀     | 不可             |
| multilingual-e5-large         | OSS  | 無料            | 優秀     | **可**           |
| nomic-embed-text-v2-moe       | OSS  | 無料            | 優秀     | **可**（ollama） |

**重要な発見**: voyage-4-nano は Apache 2.0 ライセンスで ollama からローカル実行でき、API 不要・コストゼロ。将来 API 版に切り替えてもベクトル空間が互換。

#### 軽量 Vector DB

| DB         | 永続化         | 増分更新    | 依存の軽さ | 600 件適合         |
| ---------- | -------------- | ----------- | ---------- | ------------------ |
| ChromaDB   | SQLite ベース  | upsert 対応 | やや重い   | 最適               |
| LanceDB    | Lance 形式     | 対応        | 中程度     | 適合               |
| sqlite-vec | SQLite 拡張    | 対応        | 軽い       | 適合（メンテ注意） |
| FAISS      | 手動 save/load | 自前実装    | 中程度     | 適合               |

**重要な発見**: ChromaDB が 600 件規模に最適。API がシンプルで増分更新も容易。

### 3.4 ハイブリッド検索

| 項目             | 調査結果                                                                          |
| ---------------- | --------------------------------------------------------------------------------- |
| 有効性           | 600 件でも有効。単体比で recall（漏れなさ）が 3-3.5 倍向上の報告あり              |
| 推薦融合手法     | Convex Combination（数十件のテストクエリで alpha チューニング可。RRF より高精度） |
| 全件スコアリング | **600 件なら全件ランキングも現実的** → recall 理論上 100%                         |
| 日本語 BM25      | メタデータが英語なら空白分割で十分。本文検索時は Sudachi 推薦                     |

**重要な発見**: 600 件は十分に小さいため、BM25 + Vector の**全件スコアリング**（検索ではなく全文書のランキング）が現実的。これなら recall 100% を保証でき、見落とし 0 の要件を最も確実に満たせる。

### 3.5 リランキング

| 結論      | **600 件では現時点で不要**                                                  |
| --------- | --------------------------------------------------------------------------- |
| 根拠      | BM25 + メタデータで十分な精度。小規模コーパスではリランキングの効果が限定的 |
| 将来候補  | Claude Haiku で top-20 リランク（$0.01 以下/回）が最もコスパ良好            |
| FlashRank | 軽量（4MB）だが ONNX Runtime 依存                                           |

### 3.6 日英混在トークナイザー

| 結論                   | **メタデータが英語中心なら日本語トークナイザー不要**                       |
| ---------------------- | -------------------------------------------------------------------------- |
| メタデータ検索         | unicode61（空白分割）で十分に機能する                                      |
| 本文検索を追加する場合 | 文字種境界分割 + bigram（外部依存なし、Python 標準の正規表現のみで実装可） |
| さらに精度が必要な場合 | Janome（Pure Python）または Sudachi を依存追加                             |

## 4. 設計方針の選択肢

調査結果を踏まえ、以下の 4 つの設計方針が考えられる:

### 方式 A: SQLite FTS5 単体

```
文書 → FTS5 インデックス → BM25 キーワード検索 → 候補リスト
```

| 項目         | 評価                                                     |
| ------------ | -------------------------------------------------------- |
| 外部依存     | **なし**（Python 標準ライブラリのみ）                    |
| 見落とし耐性 | 中（キーワードの言い換えに弱い）                         |
| 実装コスト   | 低                                                       |
| 適合場面     | メタデータが英語で、キーワードが明確に定義されている場合 |

### 方式 B: BM25 自前実装 + JSON 永続化

```
文書 → Python BM25 スコアリング → JSON 保存 → キーワード検索 → 候補リスト
```

| 項目         | 評価                                  |
| ------------ | ------------------------------------- |
| 外部依存     | **なし**（Python 標準ライブラリのみ） |
| 見落とし耐性 | 中（方式 A と同等）                   |
| 実装コスト   | 最低（~100 行）                       |
| 適合場面     | 最小構成で始めたい場合                |

### 方式 C: Embedding + Vector DB

```
文書 → Embedding API → ChromaDB 格納 → ベクトル類似検索 → 候補リスト
```

| 項目         | 評価                                             |
| ------------ | ------------------------------------------------ |
| 外部依存     | chromadb + embedding モデル（ollama or API）     |
| 見落とし耐性 | 高（意味的な類似性で検索）                       |
| 実装コスト   | 中                                               |
| 適合場面     | キーワードの言い換えや隠れた関連性を捉えたい場合 |

### 方式 D: ハイブリッド（BM25 + Vector）

```
文書 → FTS5 + ChromaDB に二重登録
検索 → BM25 全件スコア + Vector 全件スコア → Convex Combination → 候補リスト
```

| 項目         | 評価                                                      |
| ------------ | --------------------------------------------------------- |
| 外部依存     | chromadb + embedding モデル                               |
| 見落とし耐性 | **最高**（キーワード一致 + 意味的類似性の両方でカバー）   |
| 実装コスト   | 高                                                        |
| 適合場面     | 見落とし 0 が最優先で、600 件全件ランキングを許容する場合 |

### 方式比較まとめ

| 方式            | 外部依存 | 見落とし耐性 | 実装コスト | セマンティック検索 |
| --------------- | -------- | ------------ | ---------- | ------------------ |
| A: FTS5 単体    | なし     | 中           | 低         | 不可               |
| B: BM25 自前    | なし     | 中           | 最低       | 不可               |
| C: Vector 単体  | あり     | 高           | 中         | **可**             |
| D: ハイブリッド | あり     | **最高**     | 高         | **可**             |

## 5. 採用方針（当初案 — 2026-03-29 時点）

> **⚠️ 変更あり（2026-04-11）**: 本セクションは当初案の記録であり、最終実装とは異なる。
> v0.2.0 の品質テスト（ゴールデンセット 43 クエリ）で **ToC の検索精度が Embedding Index を上回る** ことが判明したため、
> ToC を主軸・Index を補完とするハイブリッド方式（DES-006 v2.0）に転換した。
> 最終アーキテクチャは DES-006 §2.3 および NFR-002（凍結）を参照。

### 当初採用方式: Embedding + 全文検索スクリプト（ToC 置換）

```
[インデックス構築時]
  文書メタデータ → OpenAI Embedding API → JSON に保存

[検索時]
  タスク説明文 → Embedding で候補取得（コサイン類似度）
              → AI が固有名詞を検出した場合 → 全文検索スクリプトで補完
              → AI が候補文書を Read して最終確認
```

### 採用理由

1. **Embedding 単体で十分な精度が見込める**: doc-advisor のメタデータは英語の構造化テキストであり、検索クエリ（タスク説明文）との意味的マッチングに Embedding が適している
2. **BM25 は不要**: Embedding が苦手な固有名詞・識別子の検索は、AI 自身が判断して全文検索スクリプトで補完できる。BM25 をシステムとして構築する必要がない
3. **外部依存が最小限**: OpenAI API キーのみ。chromadb、ollama、BM25 ライブラリは全て不要
4. **実装がシンプル**: 3 本の Python スクリプト（標準ライブラリのみ）で完結

### 必要なスクリプト

| スクリプト       | 役割                                               | 依存           |
| ---------------- | -------------------------------------------------- | -------------- |
| `embed_docs.py`  | 文書メタデータ → Embedding → JSON 保存（増分対応） | urllib（標準） |
| `search_docs.py` | クエリ → Embedding → コサイン類似度 → 候補リスト   | urllib（標準） |
| `grep_docs.py`   | キーワード → 全文書を文字列検索 → パスリスト       | なし（標準）   |

### 段階的検証

| Phase | やること                                   | 判断ポイント                        |
| ----- | ------------------------------------------ | ----------------------------------- |
| **1** | Embedding 単体で実装し、精度を実測         | ゴールデンセットで見落とし 0 を確認 |
| **2** | 不足があれば BM25 を追加（ハイブリッド化） | Phase 1 で漏れたケースの分析        |

### 不採用とした方式

| 方式              | 不採用理由                                             |
| ----------------- | ------------------------------------------------------ |
| BM25 単体（A/B）  | 現行 AI 検索より劣化する。セマンティック検索ができない |
| ハイブリッド（D） | Embedding 単体の精度を検証する前に導入するのは過剰設計 |
| ChromaDB          | 600 件では不要。JSON + コサイン類似度で十分            |
| ollama            | 環境依存が大きい。OpenAI API の方がシンプル            |
