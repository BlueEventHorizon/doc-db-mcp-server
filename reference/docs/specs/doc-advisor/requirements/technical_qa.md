# セマンティック検索 技術 Q&A

本文書は semantic-query の要件定義にあたり、技術選定の過程で交わされた問答を記録したものである。
初学者にとって、技術概念の理解の助けになることを意図している。

---

## Q1: Embedding は AI を使うということ？

**A**: はい。Embedding（テキストの意味を数値化する処理）は AI（機械学習モデル）でないとできない。

| 処理                              | AI なし（ルールベース） | AI あり（Embedding） |
| --------------------------------- | ----------------------- | -------------------- |
| 「ログイン」と「認証」が似ている  | わからない              | わかる               |
| 「migration」と「移行」が似ている | わからない              | わかる               |
| キーワード完全一致                | できる（BM25）          | —                    |

BM25 は「文字列が一致するか」しか見ないので、AI は不要。
Embedding は「意味が近いか」を判断するので、大量のテキストで学習した AI モデルが必要。

---

## Q2: ベクトル検索はどこが受け持つのか？

**A**: Embedding とベクトル検索は別の処理であり、担当が異なる。

```
[1. Embedding]          [2. ベクトル検索]
テキスト → ベクトル     ベクトル同士の近さを計算
   ↑                      ↑
   AI モデル               ただの数学（コサイン類似度）
```

| 処理                    | 担当                          | AI か？                             |
| ----------------------- | ----------------------------- | ----------------------------------- |
| テキスト → ベクトル変換 | Embedding モデル（OpenAI 等） | **AI**                              |
| ベクトルの保存・検索    | Vector DB や自前コード        | **AI ではない**（数学的な距離計算） |

ベクトル検索の実体は「2つの数値列の角度がどれだけ近いか」を計算しているだけ。
600 件程度なら Vector DB を使わなくても、Python の数行のコードで書ける。

---

## Q3: Embedding + Vector DB が「外部依存」とは？

**A**: 整理すると、本当に外部依存が必要なのは Embedding モデルの実行環境だけ。

| コンポーネント                | 600 件で本当に必要か                         | 代替                         |
| ----------------------------- | -------------------------------------------- | ---------------------------- |
| **Embedding モデル**          | **必要**（意味の数値化は AI にしかできない） | 代替なし                     |
| **ChromaDB（Vector DB）**     | **不要**                                     | JSON + Python で十分         |
| **HNSW 等の高速インデックス** | **不要**                                     | 600 件なら全件総当たりで一瞬 |

600 件は十分に小さいため、高速化のための Vector DB は不要。
ベクトルを JSON ファイルに保存し、検索時に全件のコサイン類似度を計算すれば済む。

---

## Q4: OpenAI の Embedding API が使えるのでは？

**A**: その通り。これにより構図がかなりシンプルになる。

| コンポーネント | 実装方法                      | 外部依存                             |
| -------------- | ----------------------------- | ------------------------------------ |
| Embedding      | OpenAI API（HTTP リクエスト） | **API キーのみ**（pip install 不要） |
| ベクトル保存   | JSON ファイル                 | なし（標準ライブラリ）               |
| ベクトル検索   | コサイン類似度（Python 数行） | なし（標準ライブラリ）               |

API 呼び出しは Python 標準ライブラリの `urllib` だけで書ける。
600 件のインデックス全体再構築でも約 $0.01（約 1.5 円）。

つまり、**外部依存は「OpenAI API キー」だけ**で Embedding 検索が実現できる。
chromadb も ollama も不要。

## Q5: Embedding は単語のベクトルなのでセマンティックではない？

**A**: 2013 年頃の Embedding（Word2Vec 等）は確かに単語単位だった。しかし現代の Embedding は**文全体のベクトル**を生成する。

### Embedding の世代による違い

| 世代           | 代表例                  | 単位         | セマンティックか                                           |
| -------------- | ----------------------- | ------------ | ---------------------------------------------------------- |
| **2013年**     | Word2Vec, GloVe         | 単語         | **No**。単語の共起パターンを学習しただけ                   |
| **2026年現在** | OpenAI text-embedding-3 | **文・段落** | **Yes**。Transformer が文脈を理解して 1 つのベクトルを生成 |

現代のモデルは文全体を読み、各単語の関係性・文脈を考慮して 1 つのベクトルを出力する。
だから「データ移行の実装」と「Staged Version Migration Specification」が単語として一致しなくても、近いベクトルになる。

### ただし限界はある

「セマンティック」と言っても、人間のような深い理解ではない:

- 学習データの統計パターンに基づく類似性判定
- ドメイン固有の知識（doc-advisor の文脈等）は持っていない
- 「なぜ関連するか」の推論はしていない。ただ「近い」と判定するだけ

正確に言うと: **Embedding は「浅いセマンティック検索」。意味の「方向」は捉えるが、「深い推論」はしない。**

## Q6: BM25 と Embedding の両方が必要か？

**A**: doc-advisor の条件（600 件・英語メタデータ・タスク説明文での検索）を考えると、**Embedding 単体でも十分な精度が出る可能性が高い**。

### Embedding 単体で漏れるケース

- `doc_structure.yaml` のような識別子での検索
- `SHA-256`、`FNC-002` のような技術固有名詞や ID での検索

### ただし doc-advisor では

検索は「タスクの説明文 → 関連文書の発見」であり、ID やファイルパスで検索するユースケースは想定されていない。タスク説明文 vs 英語メタデータの組み合わせなら、Embedding のセマンティック理解が活きる。

### 段階的アプローチ

| Phase | やること                                       |
| ----- | ---------------------------------------------- |
| **1** | Embedding 単体で実装し、精度を実測する         |
| **2** | 不足があれば BM25 を追加してハイブリッド化する |

いきなりハイブリッドを組むより、Embedding 単体でどこまでいけるか試す方が合理的。

## Q7: クエリに固有名詞がある場合はどうする？

**A**: AI（Claude）がクエリを受け取った時点で「これは固有名詞だ」とわかるので、全文検索スクリプトで補完すればよい。

| クエリの中身                             | 担当                | 方法                  |
| ---------------------------------------- | ------------------- | --------------------- |
| 意味的な内容（「検索機能を改善したい」） | **Embedding**       | ベクトル類似度        |
| 固有名詞・識別子（`doc_structure.yaml`） | **AI + スクリプト** | 全文検索（grep 相当） |

つまり **BM25 というシステムを別途構築しなくても、AI 自身が固有名詞の検索を担える**。Embedding が苦手なケースを AI がカバーする形。

### 当初想定のアーキテクチャ（2026-03-29 時点）

> **⚠️ 変更あり（2026-04-11）**: 下記は当初案であり、最終実装とは異なる。
> v0.2.0 の品質テストで ToC の検索精度が Embedding Index を上回ったため、
> **ToC を廃止せず主軸として維持し、Index を補完に使うハイブリッド方式**（DES-006 v2.0）に転換した。

```
[インデックス構築時]
  文書メタデータ → OpenAI Embedding API → JSON に保存

[検索時（当初案）]
  タスク説明文 → Embedding で候補取得（コサイン類似度）
              → AI が固有名詞を検出した場合 → 全文検索スクリプトで補完
              → AI が候補文書を Read して最終確認
```

外部依存は OpenAI API キーのみ。BM25 も Vector DB も不要。
