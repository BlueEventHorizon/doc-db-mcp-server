# REQ-006: doc-db plugin 要件定義書

## 概要

doc-db は、プロジェクトのドキュメントを **見出し単位 chunk の Hybrid 検索（Embedding + Lexical）と LLM Rerank** で検索する独立 plugin である。実装は **doc-advisor v0.2.x の embedding 機構（`plugins/doc-advisor/scripts/embed_docs.py` / `embedding_api.py` / `search_docs.py` / `create_checksums.py`）を出発点とし、その上に chunk 化・Lexical 検索・Hybrid 統合・LLM Rerank・ゴールデンセット評価を追加する** 。

doc-advisor は本 plugin の品質比較ベースラインとして不変に保つ。doc-db の存在・利用は doc-advisor 本体の挙動を変更しない。将来 doc-db 方式が ToC 方式・現行 Embedding 方式を上回ると判断された場合は、doc-advisor を doc-db で置き換える可能性がある。

## 前提条件

| ID     | 要件                                                                                                                                                                                       |
| ------ | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ |
| PRE-01 | `.doc_structure.yaml` がプロジェクトルートに存在する（doc-advisor と共有。`root_dirs` / `target_glob` / `exclude` / `doc_types_map`）                                                      |
| PRE-02 | OpenAI API key が環境変数で利用可能である。参照規約は **`OPENAI_API_DOCDB_KEY` を優先、未設定時のみ `OPENAI_API_KEY` をフォールバック**（FNC-004 / DES-007 統一 API KEY 参照規約に準ずる） |

## 機能要件サマリ

| ID      | 概要                                                  |
| ------- | ----------------------------------------------------- |
| FNC-005 | 見出し単位 chunk の Multi-vector Embedding Index 生成 |
| FNC-006 | Hybrid 検索（Embedding + Lexical）と LLM Rerank       |
| FNC-007 | ゴールデンセットによる検索品質評価                    |

## 非機能要件サマリ

| ID      | 概要                                                 |
| ------- | ---------------------------------------------------- |
| NFR-004 | ToC 方式と同等以上の recall を達成する               |
| NFR-005 | 標準ライブラリのみで動作し、pip install を要求しない |

## 受け入れ条件

| ID    | 条件                                                                                                                                 |
| ----- | ------------------------------------------------------------------------------------------------------------------------------------ |
| AC-01 | doc-db plugin は doc-advisor 本体と独立して動作し、doc-db の利用が doc-advisor 本体の挙動を変更しない（並行運用 / ベースライン保護） |
| AC-02 | 同一ゴールデンセット（`meta/test_docs/`）で ToC 方式・現行 Embedding 方式・doc-db 方式の recall を比較できる                         |
| AC-03 | NFR-004 で定義する recall 同等基準を満たす（具体的な測定契約は評価計画で確定）                                                       |
| AC-04 | false negative の傾向が分類されたレポートが生成される                                                                                |
| AC-05 | 標準ライブラリのみで動作する（外部 pip パッケージを必要としない）                                                                    |

## 関連要件

- 子要件: FNC-005（Index 生成）, FNC-006（検索）, FNC-007（評価）, NFR-004（検索品質）, NFR-005（配布制約）
- 参照実装（出発点）: `plugins/doc-advisor/scripts/` 配下の embedding 機構

## 用語定義

| 用語             | 定義                                                                                                               |
| ---------------- | ------------------------------------------------------------------------------------------------------------------ |
| chunk            | 検索の最小単位。Markdown の見出し（`#` 以下の section）を境界として 1 chunk を構成する                             |
| Hybrid 検索      | Embedding 検索（意味類似度）と Lexical 検索（語彙一致）を組み合わせ、両スコアを統合する手法                        |
| LLM Rerank       | 取得した候補 chunk を LLM に渡し、関連度順に並び替える後処理                                                       |
| ゴールデンセット | 期待する検索結果が定義された評価用クエリ集合（`meta/test_docs/`）                                                  |
| doc_type         | `.doc_structure.yaml` の `doc_types_map` で定義される文書種別（`requirement` / `design` / `plan` / 将来 `adr` 等） |
