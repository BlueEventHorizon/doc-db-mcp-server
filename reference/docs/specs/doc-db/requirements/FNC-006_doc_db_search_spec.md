# FNC-006: doc-db 検索 要件定義書

## 概要

ユーザーのクエリに対し、Embedding 検索と Lexical 検索を組み合わせた **Hybrid 検索** で候補を取得し、**LLM Rerank** で並び替えた最終結果を返す。実装は doc-advisor の検索機構（`plugins/doc-advisor/scripts/search_docs.py`）を出発点として拡張する。

## 前提条件

- Index が生成済みである（FNC-005 完了済み）
- REQ-006 PRE-02 に準ずる（OpenAI API key の参照規約は FNC-004 / DES-007 に準拠）

## 要件一覧

### doc-advisor からの継承

以下は doc-advisor の検索機構に準拠する。本要件では再定義しない。

- クエリの Embedding 化（Index と同じモデル）
- Embedding 検索（cosine similarity 等の意味的類似度に基づく順位付け）
- 検索開始時の Index 鮮度確認・差分自動再生成（`schema_version` / モデル / 次元数の整合性チェック含む）
- 標準的なエラーハンドリング（Index 不在 / API 呼び出し失敗 / model mismatch 時の全体再生成案内）

### doc-db で追加・変更する要件

#### 操作

| ID    | 要件                                                                                                                               |
| ----- | ---------------------------------------------------------------------------------------------------------------------------------- |
| OP-01 | ユーザーは自然文または ID・固有名詞を含むクエリで検索できる                                                                        |
| OP-02 | ユーザーは取得結果数の上限（top-N）を指定できる                                                                                    |
| OP-03 | ユーザーは検索方式（embedding のみ / lexical のみ / hybrid / hybrid + rerank）を選択できる                                         |
| OP-04 | specs 検索のデフォルト doc_type は `requirement` / `design`（既存 `query-specs` インターフェースと互換）。複数 doc_type を指定可能 |
| OP-05 | rules 検索は doc_type 概念を持たず、rules 配下を単一カテゴリとして検索する                                                         |

#### 検索本体

| ID     | 要件                                                                                                                                                      |
| ------ | --------------------------------------------------------------------------------------------------------------------------------------------------------- |
| EMB-01 | Multi-vector（chunk 単位 / field 単位）を持つ文書は、各ベクトルのスコアを集約してランキングに使用する（集約方式は設計書）                                 |
| LEX-01 | クエリ語の語彙一致に基づいて chunk スコアを算出する（スコアリング方式は BM25、詳細は設計書）。ID（例: `FNC-001`）や固有名詞の完全一致を高スコアとして扱う |
| SC-01  | Embedding スコアと Lexical スコアを統合して候補をソートする（統合方式は設計書、設定で切り替え可能）                                                       |
| RR-01  | 統合スコア上位の候補を LLM Rerank API に渡し、関連度順に並び替える。LLM Rerank は ON/OFF を切り替えできる                                                 |
| RR-02  | LLM Rerank が失敗した場合は rerank をスキップし、統合スコア順の結果を返す（fallback）                                                                     |

#### grep 補完（SKILL レイヤー）

| ID     | 要件                                                                                                                                  |
| ------ | ------------------------------------------------------------------------------------------------------------------------------------- |
| GRP-01 | SKILL レイヤーはクエリに ID・固有名詞が含まれる場合、Hybrid 検索に加えてファイルシステム上の grep（全文検索）を実行し、結果を統合する |
| GRP-02 | grep はファイルシステムを直接検索するため、Index が incomplete（embedding 失敗 chunk あり）の場合でも漏れを防止する                   |
| GRP-03 | grep の対象ファイルは `.doc_structure.yaml` の `root_dirs` / `target_glob` / `exclude` に従う（Hybrid 検索と同一スコープ）            |

> doc-advisor の `query_index_workflow.md` Step 2 と同等の設計。Embedding が苦手とする「IDの完全一致」を grep で補完し、false negative を最小化する。

#### 出力

| ID     | 要件                                                                                                                   |
| ------ | ---------------------------------------------------------------------------------------------------------------------- |
| OUT-01 | 結果は文書パス・該当 chunk の見出し階層・該当 chunk テキスト・スコア・スコア内訳（embedding / lexical / rerank）を含む |
| OUT-02 | なぜヒットしたか（どの chunk のどのスコアで残ったか）を結果に提示する                                                  |

> 類似度尺度・探索アルゴリズム・スコア統合式・Lexical 単語分割方式・LLM Rerank に渡す候補数上限・入力バリデーション規則（top-N 範囲・クエリ文字数）は設計書で確定する。
